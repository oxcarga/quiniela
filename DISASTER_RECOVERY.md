# Disaster Recovery — running `repair-scores.ts` against production

This plan covers the one operation we're nervous about: running
[`scripts/repair-scores.ts`](scripts/repair-scores.ts) against the **production**
Firestore database to backfill `exactCount` (and re-derive `totalScore`) and
rebuild the leaderboard.

Read the whole thing once before you run anything.

---

## 1. What the script actually changes (blast radius)

`repair-scores.ts` writes to **only three places**:

| Target | Field(s) written | Notes |
|---|---|---|
| `users/{uid}` | `totalScore`, `exactCount` | Recomputed from the user's predictions |
| `predictions/{uid}/matches/{doc}` | `exactHit` (boolean) | Backfilled per prediction |
| `leaderboard/current` | whole doc (`rankings`, `updatedAt`) | Rebuilt from `users` |

It does **NOT** touch, delete, or rewrite:

- `pointsEarned` on any prediction (**the source of truth**)
- `predictedHomeGoals` / `predictedAwayGoals` / `boosted` / any prediction inputs
- `matches/*` (fixtures + results)
- Auth users, `loginCodes`, or anything else

### Why this matters

Every value the script writes is **derived deterministically** from data it does
not modify (`pointsEarned` and the prediction inputs). That gives us two strong
safety properties:

1. **Idempotent** — running it twice produces the same result. A partial/failed
   run is fixed by simply running it again.
2. **Reconstructible** — as long as `pointsEarned` and the prediction docs are
   intact, the *correct* `totalScore`/`exactCount`/`exactHit` can always be
   regenerated by re-running. There is no "lost" state to recover, only derived
   state to recompute.

So the realistic failure modes are narrow (see §5). True data loss would require
a bug that writes to fields the script isn't supposed to — which is why the
pre-flight backup in §3 still matters.

---

## 2. Known gap: there is no restore-to-prod script

- [`scripts/export-db.ts`](scripts/export-db.ts) **can** read production (with a
  service account) and write a full JSON snapshot to the repo root.
- [`scripts/import-db.ts`](scripts/import-db.ts) **refuses to run against prod** —
  it exits unless `FIRESTORE_EMULATOR_HOST` is set. It is emulator-only by design.

**Implication:** a full JSON export is our backup, but restoring it is not a
one-command operation today. Two restore routes exist, in order of preference:

- **(A) GCP-managed Point-in-Time Recovery (PITR) / scheduled backups** — the
  safest, operates at the database level, no custom code. Strongly recommended to
  enable *before* the run (see §6).
- **(B) A purpose-built targeted restore** that rewrites only the three affected
  targets from the pre-run export (see §5, scenario 3). Small and safe because the
  field set is tiny.

## 3. Pre-flight checklist (do all of these before running)

1. **Confirm the target project.** The scripts read the project id from
   `.firebaserc` (`projects.default` → `quiniela-ee895`). Verify you intend to
   write to that project and that `./functions/service-account.json` belongs to it.

2. **Take a full backup (primary safety net).** Snapshot prod to a JSON file in
   the repo root:

   ```bash
   GOOGLE_APPLICATION_CREDENTIALS=./functions/service-account.json \
     npx tsx ./scripts/export-db.ts
   ```

   Note the output filename: `firestore-export-<timestamp>.json`. **Keep it.**
   Copy it somewhere outside the repo too (it contains user data — handle
   accordingly, don't commit it).

3. **Verify the backup loads into the emulator** (proves it's a usable restore
   source, not a corrupt file):

   ```bash
   # emulators running: npm run emulators
   FIRESTORE_EMULATOR_HOST=localhost:8080 FIREBASE_AUTH_EMULATOR_HOST=localhost:9099 \
     npx tsx scripts/import-db.ts firestore-export-<timestamp>.json --clear
   ```

4. **Rehearse the exact run locally first.** Import the prod snapshot into the
   emulator (step 3 already did this), then run the repair against the emulator
   and confirm the output looks sane (scores mostly `OK`, `exactCount` populated,
   leaderboard order correct):

   ```bash
   FIRESTORE_EMULATOR_HOST=localhost:8080 npx tsx scripts/repair-scores.ts
   ```

   This is the single most valuable safeguard: prod and the emulator are running
   the *same* script against the *same* data. If it behaves locally, it will
   behave in prod.

5. **Pick a low-traffic window.** The `updateLeaderboard` trigger fires on every
   `users/{uid}` write, so the repair's user updates will each rebuild the
   leaderboard. Fine, but quieter is better.

6. **Confirm functions are already deployed** (`npm run func:deploy`) so the live
   `scoreMatch`/`rebuildLeaderboard` match the script's logic before backfilling.

## 4. Execution

```bash
GOOGLE_APPLICATION_CREDENTIALS=./functions/service-account.json \
  npx tsx scripts/repair-scores.ts
```

Watch the output. Every user prints a line:

```
<uid>  totalScore: OK | A → B ✓ FIXED   exactCount: 0 → N ✓ FIXED
```

Expected on first prod run: `totalScore` mostly `OK` (already correct), most
`exactCount` going `0 → N` (the backfill). It ends with
`Committing X update(s)... Leaderboard rebuilt. Done.`

If it errors mid-way, that's fine — see §5 scenario 1.

## 5. Recovery procedures by scenario

### Scenario 1 — Run failed / crashed part-way through

Some users updated, others not; leaderboard maybe stale.

**Fix:** just run it again. It's idempotent — it recomputes from `pointsEarned`
and only writes deltas, so a second run completes whatever the first missed and
leaves correct state. No restore needed.

### Scenario 2 — Run finished but numbers look wrong (logic bug in the script)

`pointsEarned` (source of truth) is untouched, so the data isn't lost — it's just
that the derived values are wrong.

**Fix:** correct the script, re-run locally against the emulator copy of prod to
verify, then re-run against prod. Because the script is deterministic, the
corrected run overwrites the bad derived values with right ones.

If the leaderboard cache is wrong but `users` are right, force a rebuild by
touching any user doc (fires `updateLeaderboard`) or re-run the repair (it rebuilds
at the end).

### Scenario 3 — Wrong fields were corrupted / wrong project / `pointsEarned` damaged

This is the genuine "restore" case (should not happen given §1, but plan for it).

**Option A — GCP PITR (preferred, if enabled — see §6).** Recover the database (or
read a past timestamp) from before the run. Database-level, no app code.

**Option B — Targeted restore from the pre-run export.** Write a small one-off
restore script (we don't keep one because `import-db.ts` is emulator-only) that,
from `firestore-export-<timestamp>.json`, rewrites **only** the affected fields:

- For each `users/{uid}`: set `totalScore` and `exactCount` back to the export's
  values.
- For each `predictions/{uid}/matches/{doc}`: set `exactHit` back (or delete it).
- Overwrite `leaderboard/current` with the export's `rankings`.

Keep its scope to those fields only — never bulk-overwrite whole user docs (you'd
clobber any legitimate writes made since the export). The export format is the
nested `_subcollections` shape produced by `export-db.ts`; reuse its
`reviveTimestamps`/`flatten` helpers as a starting point.

> If you reach this scenario without PITR enabled, the pre-run export from §3
> step 2 is the only source of truth for the old values. Do not skip it.

### Scenario 4 — Leaderboard order looks wrong but scores are correct

Likely the `exactCount` field isn't populated yet (e.g. data re-imported after the
run) or a stale `leaderboard/current`. Re-run the repair, or confirm `exactCount`
exists on `users/*`. Not a data-loss event.

## 6. Recommended hardening (do this before the prod run if possible)

1. **Enable Point-in-Time Recovery** on the Firestore database (7-day recovery
   window). This converts Scenario 3 from "hope the export is good" into a
   database-level rollback. Verify exact syntax against current `gcloud` docs:

   ```bash
   gcloud firestore databases update --database='(default)' \
     --enable-pitr --project=quiniela-ee895
   ```

2. **Take a managed backup** immediately before the run (independent of the JSON
   export), e.g. an on-demand backup or an export to a GCS bucket:

   ```bash
   gcloud firestore export gs://<your-backup-bucket>/pre-repair-$(date +%Y%m%d) \
     --project=quiniela-ee895
   ```

3. **Consider a `--dry-run` flag** for `repair-scores.ts` that logs every intended
   change without committing. Today the script prints per-user diffs but commits at
   the end; a true dry-run would let you eyeball the full prod diff with zero
   writes. Cheap to add and worth it.

---

## 7. TL;DR runbook

1. `export-db.ts` against prod → keep the JSON. **(backup)**
2. Import that JSON into the emulator, run `repair-scores.ts` locally → verify. **(rehearsal)**
3. (Recommended) Enable PITR + managed backup. **(safety net)**
4. Run `repair-scores.ts` against prod in a quiet window. **(execute)**
5. Spot-check a few users + the leaderboard order. **(verify)**
6. If anything's off: re-run (idempotent) first; restore from PITR/export only if
   a non-derived field was damaged.
